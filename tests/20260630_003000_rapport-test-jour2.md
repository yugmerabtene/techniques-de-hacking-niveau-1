## Problèmes identifiés

1. **Source path `../../..`** : `lab_j2.sh` et `recon.sh` utilisent `../../..` pour sourcer `env.sh` (déjà corrigé dans les scripts Day 1). N'affecte pas l'exécution.

2. **Versions de services** : Le conteneur vsftpd expose `nginx` au lieu d'`Apache`, et `ssh` uniquement via IP interne. Les cibles vulnérables (vsftpd 2.3.4, Samba 3.0.20) sont présentes.

3. **Reverse shell LHOST** : Les exploits Metasploit nécessitent `LHOST=172.17.0.1` (passerelle Docker) et non `127.0.0.1` pour que le reverse shell atteigne l'hôte depuis le conteneur.

4. **One-shot bind shell vsftpd** : Le port 6200 est un one-shot — une fois utilisé, il faut re-déclencher la backdoor via `printf "user :)\r\n..." | nc`.

5. **DNS dans conteneurs** : Les conteneurs minimalistes n'ont pas de résolution DNS nativement. Le DNS spoofing BetterCap nécessite une configuration DNS dans le conteneur cible.
